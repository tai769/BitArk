## 2. 当前工程背景与进度存档

我们正在开发 **BitArk**——一个追求百亿级高并发的纯内存已读服务中间件（RoaringBitmap + WAL）。目前在攻坚 Master-Slave 的心跳保活与数据同步协调模块。

截止本文件生成的那一刻，我们已经重构完了 `ReplicationTrackerImpl` 的完整核心，取得了以下成绩：
1. **解决并发竞态**：将 `SlaveState` 设计成了全 `final` 的不可变快照对象；使用了 `ConcurrentHashMap` 提供的带条件两参数 `remove(key, oldVal)`（CAS），以及 `compute()` 来安全替换心跳状态，做到了无锁并发；确保并优化了锁内避免做任何 IO 操作。
2. **遵循依赖倒置**：理清了 Tracker（只管心跳和状态）和 Writer（只管写盘）是两条独立的线。为了让 Tracker 能在每次接到 `onHeartbeat` 时判断从节点的滞后程度（Lag），我们构造器注入了 `Supplier<WalCheckpoint> walCheckpointSupplier` 而不是直接依赖大而全的写日志类。
3. **实现 ISR 状态机**：`getMinAckLsn()` 现在被严格约束为只会去计算 `status == ReplicaStatus.ISR` 且未超时的节点，避免了刚通过 `registerAck` 注册归零的断层节点 (`OBSERVER`) 将主节点的 WAL GC（日志回收）给无故锁死。

## 3. 下一步等待你引导秋水的断点

我们在之前的对话结束时，留下了三个亟待解决的工程问题（或隐藏的“雷”）。
**请从第一条开始，向秋水发问，继续我们的工程推导：**

### 雷区一：跨文件的 Lag 计算方式错误 (最致命)
目前在 `onHeartbeat` 中，判断 Lag 的代码写的是 `master.getSegmentOffset() - slave.getOffset() > Lag`。
**引导方向**：如果跨了 segment（即跨文件了，比如 master 落到了 segment 3, offset 500 的位置，而 slave 还在 segment 1, offset 8000 缓慢追赶），这个单纯的偏移量一减，直接变成了负数（-7500）。此时这个其实已经落后了两个大文件的 Slave 竟然会被误判为 lag 在阈值内，从而错误晋升为 `ISR`！你需要反问秋水，如果每个文件的物理大小是固定的（比如 1GB），如何把 `(segmentIndex, offset)` 这个二元组转换为一个单调递增的绝对字节序号进行运算？

### 雷区二：事件的 推/拉 模型差异
Tracker 里面目前已经能判断出哪些 Slave 被降级为 `OUT_OF_SYNC` 从而 `needsFullSync = true`，但是谁去执行全量同步？
**引导方向**：上阶段我们用 `Supplier` “主动拉”解决依赖倒置，但这回相反，是需要“向外推”事件告诉 `RecoveryCoordinator`。引导秋水思考类似 `Consumer<T>` 或者观察者模式的运用。

### 雷区三：Sender 的遍历困局
看看目前的 `HttpReplicationSender.sendRead` 里的代码，目前只是硬编码往配置文件里的**唯一一个** `slaveUrl` 发数据。
**引导方向**：真实集群里有 N 个 Slave，Master 此时该怎么去向 Tracker 要当前的存活分发列表，然后把单播并发成广播（不阻塞主业务线程）？
